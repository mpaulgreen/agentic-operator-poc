# Sprint 2 Gap Analysis: `designing-operator-api` Skill vs operator-sdk

## Types (Workflow A+B)

The SDK generates **stub types** with a `Foo` example field. The skill generates **real types** from user requirements.

| Aspect | SDK | Skill |
|--------|-----|-------|
| Spec fields | `Foo string` (example) | User-specified with validation markers |
| Validation markers | 0 | 9+ (Minimum, Maximum, Enum, Default, Pattern) |
| Print columns | 0 | 4+ (Phase, Ready, Version, Age) |
| Conditions in Status | Not present | `[]metav1.Condition` with patchStrategy |
| Default values | 0 | 3-5 per CRD |
| Nested types | None | StorageSpec, BackupSpec, etc. |
| Embedded K8s types | None | corev1.ResourceRequirements |
| Root markers | Match | Match (+kubebuilder:object:root, +subresource:status) |

**Assessment**: The skill produces production-ready types while the SDK produces stubs that need manual work. The root type markers and package structure match exactly.
